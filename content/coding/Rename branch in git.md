#prog
Коротко: переименовываем локальную ветку, пушим её под новым именем, удаляем старую на GitHub и привязываем апстрим.

## Вариант 1 — переименовать текущую ветку A → B (рекомендуется)

```bash
# 0) Убедитесь, что всё закоммичено
git status

# 1) Переименовать текущую ветку A в B (вы на A)
git branch -m B

# 2) Запушить новую ветку и привязать апстрим
git push -u origin B

# 3) Удалить старую ветку A на удалённом репо (GitHub)
git push origin --delete A   # эквивалентно: git push origin :A
```

Проверить:

```bash
git branch -vv
git ls-remote --heads origin
```

> Если на GitHub ветка **A** была защищённой или “default branch”, сначала временно смените _Default branch_ в Settings → Branches, снимите защиту, удалите ветку, затем верните защиту/дефолт на **B**.

---

## Вариант 2 — создать новую ветку B от A и потом удалить A

```bash
# 1) От текущей A ответвляемся
git checkout -b B

# 2) Пушим и привязываем
git push -u origin B

# 3) Вернёмся (необязательно) и удалим локально A
git branch -D A

# 4) Удалим удалённую A
git push origin --delete A
```

---

## Если вы СНАЧАЛА переименовали ветку на GitHub (через UI)

GitHub умеет делать редиректы со старого имени:

```bash
# Обновить локальную
git fetch origin
git branch -m A B
git branch -u origin/B B
git remote set-head origin -a
```

---

## Частые подводные камни

- **Protected / default branch**: не удалится, пока не снимете защиту/не смените default.
    
- **Открытые PRы в A**: перенаправьте на **B** (GitHub обычно подсказывает).
    
- **CI/интеграции**: обновите ссылки на ветку A → B (workflow триггеры, баджи, автодеплой).
    
- **Ветка B уже существует на origin**: решите конфликт (переименуйте иначе или слейте/ребейзнитесь).
