### Архитектура
- Централизованная, репа на сервере
- Доступ к репозиторию через сервер Apache либо daemon `svnserve`. К *удаленному* используем связки ssh+svn либо https+svn

Клиент SVN не только осуществляет передачу данных с репозитория клиенту и обртано. Он еще осуществляет управление локальной копией файлов, которую модифицирует разработчик. 

Репозиторий – набор файлов проекта в иерархическом виде.

Фиксация изменений `svn commit` повышает версию репозитория на 1. В один коммит стоит помещать логически законеченное обноваление функционала, или, например, исправление одного дефекта.

Ораганизация файлов: существуюбт каталоги проектов, внутри которых размещены директории, предназначенные для организации работы с разными версиями проекта.

В каталоге (ветке) *trunk* происходит основной процесс разработки.  *branch – хранение модификаий продукта*, *tags –функционально целостные изменения*.

#### Команды 
* `svn update`
* `svn commit`
* `svn (add, delete, copy, move, mkdir`
* `svn revert`
### конфликты содержимого файлов
при `svn update` (как например при `git fetch`) возникают конфликты
их фиксим вот так:
- (e) edit — изменить файл в редакторе
- (df) diff-full — диффнуть
- (r) resolved — подтвердить, что конфликт резрешен в текущй версии файла
- (dc) display-conflict — показать все конфликты
- ==(mc) mine-conflict, (tc) theirs-conflict – подтвердить мои (или из репозитория) версии для всех конфликтов==
- ==(mf) mine-full, (tf) theirs-full — подтвердить свои версии==
- (p) postpone — отлождить на потом

Для diff ныне используют ide в которых уже все краисво показывается.

Также есть слияние `svn merge`. буквальный аналог `git merge`
